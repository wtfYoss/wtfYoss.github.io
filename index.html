1.1
Árboles de Expresiones
Representación jerárquica
+

Un árbol de expresiones (o árbol sintáctico abstracto, AST) es una estructura jerárquica en forma de árbol que representa la estructura de una expresión matemática o de un programa. Los nodos internos son operadores y las hojas son operandos (valores o variables).

Características

  • Los nodos hoja representan operandos (números, variables)
  • Los nodos internos representan operadores (+, -, *, /)
  • La raíz del árbol es el operador de menor precedencia
  • La evaluación se realiza de abajo hacia arriba (post-orden)

Ejemplo — expresión: (a + b) * c

* / \ + c / \ a b Evaluación post-orden: a b + c *

Tipos de recorridos

  • Pre-orden: raíz → izquierdo → derecho (notación prefija)
  • In-orden: izquierdo → raíz → derecho (notación infija)
  • Post-orden: izquierdo → derecho → raíz (notación postfija)
AST compiladores análisis sintáctico árbol binario
1.2
Acciones Semánticas
Del analizador sintáctico
+

Las acciones semánticas son fragmentos de código que se asocian a las reglas de producción de una gramática. Se ejecutan cuando el analizador sintáctico reconoce (reduce) una producción. Son el puente entre el análisis sintáctico y el semántico.

¿Para qué sirven?

  • Construir el árbol sintáctico abstracto (AST)
  • Verificar tipos y declaraciones de variables
  • Generar código intermedio
  • Actualizar la tabla de símbolos

Ejemplo — gramática con acción

E → E '+' T { E.val = E1.val + T.val } E → T { E.val = T.val } T → num { T.val = num.lexval }

Tipos de atributos

  • Sintetizados: se propagan de hijos a padres
  • Heredados: se propagan de padres a hijos
  • Definición dirigida por sintaxis (SDD): conjunto de atributos + reglas
SDD atributos sintetizados heredados
1.3
Comprobación de Tipos
En expresiones y operadores
+

La comprobación de tipos (type checking) verifica que cada operación se aplique a operandos del tipo adecuado. Es una de las tareas más importantes del analizador semántico para garantizar la coherencia del programa.

Tipos de verificación

  • Estática: se realiza en tiempo de compilación (C, Java, Pascal)
  • Dinámica: se realiza en tiempo de ejecución (Python, JS)
  • Fuerte: no permite operaciones entre tipos incompatibles
  • Débil: permite conversiones implícitas automáticas

Conversión de tipos (casting)

  • Implícita (coerción): el compilador convierte automáticamente
  • Explícita: el programador indica la conversión manualmente

Ejemplo — reglas de tipo

int + int → int ✓ int + float → float ✓ (coerción) int + string→ ERROR ✗ (tipo incompatible) char + int → int ✓ (coerción ASCII)
type checking coerción casting semántico
1.4
Pila Semántica
En un analizador sintáctico
+

La pila semántica es una estructura de datos paralela a la pila del analizador sintáctico. Almacena atributos y valores semánticos de los símbolos que están en la pila del parser, permitiendo que las acciones semánticas accedan a la información necesaria.

Funcionamiento

  • Cada entrada en la pila del parser tiene una entrada en la pila semántica
  • Al hacer shift: se empuja el valor del token actual
  • Al hacer reduce: se extraen los valores de los hijos y se calcula el del padre
  • Es fundamental en parsers LALR(1) y SLR(1)

Ejemplo de ejecución — 3 + 5

PILA SINTÁCTICA PILA SEMÁNTICA ENTRADA $ $ 3 + 5 $ $ num $ [3] + 5 $ $ E $ [3] + 5 $ $ E + $ [3,-] 5 $ $ E + num $ [3,-,5] $ $ E + T $ [3,-,5] $ $ E $ [8] $ ← 3+5=8
LALR SLR shift-reduce atributos
1.5
Esquema de Traducción
Reglas semánticas dirigidas por sintaxis
+

Un esquema de traducción (SDT - Syntax Directed Translation) es una gramática con acciones semánticas incrustadas en los cuerpos de las producciones. A diferencia de las SDD, indica el orden exacto en que deben ejecutarse las acciones.

Diferencia SDD vs SDT

  • SDD: especifica qué calcular, no cuándo
  • SDT: especifica qué calcular y en qué momento exacto
  • Las acciones se incrustan entre los símbolos de la producción

Ejemplo — traducción a postfija

E → E₁ '+' T { print('+') } E → T T → T₁ '*' F { print('*') } T → F F → '(' E ')' F → num { print(num.val) } Entrada: 3 + 4 * 5 Salida: 3 4 5 * +

Tipos de SDT

  • SDT con atrib. sintetizados: acciones al final de producción
  • SDT con atrib. heredados: acciones en medio de la producción
SDT postfija traducción gramática
1.6
Tabla de Símbolos
Y tabla de direcciones
+

La tabla de símbolos es una estructura de datos central en el compilador que almacena información sobre cada identificador declarado en el programa. Es utilizada por todas las fases del compilador.

Información almacenada

  • Nombre del identificador (lexema)
  • Tipo de dato (int, float, string, etc.)
  • Dirección de memoria o número de registro
  • Ámbito (scope) y nivel de anidamiento
  • Número de parámetros (si es función)

Ejemplo — tabla de símbolos

| Nombre | Tipo | Dir | Ámbito | Valor | |--------|-------|------|--------|-------| | x | int | 0x01 | global | 10 | | y | float | 0x02 | local | 3.14 | | suma | func | 0x10 | global | — |

Implementaciones comunes

  • Tabla hash: acceso O(1) promedio
  • Árbol BST: ordenado, acceso O(log n)
  • Lista encadenada: simple pero O(n) búsqueda
hash table scope identificadores memoria
1.7
Errores Semánticos
Detección y recuperación
+

Los errores semánticos ocurren cuando el código es sintácticamente correcto pero carece de sentido lógico. El compilador los detecta durante el análisis semántico al verificar restricciones que la gramática no puede capturar.

Tipos de errores semánticos

  • Variable no declarada: uso de identificador sin declarar
  • Variable ya declarada: redeclaración en el mismo ámbito
  • Incompatibilidad de tipos: int = string, por ejemplo
  • Número incorrecto de argumentos: en llamada a función
  • Uso fuera de ámbito: acceder a variable de otro scope
  • Valor fuera de rango: overflow en asignación

Estrategias de recuperación

  • Modo pánico: descarta tokens hasta símbolo de sincronización
  • Corrección de errores: intenta continuar asumiendo tipo compatible
  • Reporte múltiple: reporta todos los errores posibles antes de detener

Ejemplo

int x = 5; float y = x + "hola"; // ERROR: tipo incompatible z = x + 1; // ERROR: 'z' no declarada int x = 10; // ERROR: redeclaración de 'x'
errores recuperación scope diagnóstico

2.1
Notaciones
Prefija, Infija, Postfija
+

Las notaciones son formas distintas de escribir expresiones matemáticas y lógicas según la posición del operador respecto a sus operandos. Son fundamentales para la generación de código intermedio.

Notación Infija (2.1.2)

  • Es la notación habitual: operador entre operandos
  • Requiere paréntesis para indicar precedencia
  • Ejemplo: A + B * C

Notación Prefija / Polaca (2.1.1)

  • El operador aparece ANTES que los operandos
  • No requiere paréntesis, la precedencia es implícita
  • Ejemplo: + A * B C (A + B*C)

Notación Postfija / Polaca Inversa (2.2.3)

  • El operador aparece DESPUÉS de los operandos
  • Ideal para evaluación con pilas (máquinas de pila)
  • Ejemplo: A B C * + (A + B*C)

Tabla comparativa

Expresión: (A + B) * (C - D) Infija: (A + B) * (C - D) Prefija: * + A B - C D Postfija: A B + C D - *
infija prefija postfija Polaca Inversa
2.2
Representaciones de Código Intermedio
Polaca, P, Triples, Cuádruplos
+

Las representaciones de código intermedio son estructuras que sirven como puente entre el código fuente de alto nivel y el código máquina final. Independizan el compilador de la arquitectura de la máquina destino.

2.2.1 Notación Polaca (RPN)

  • Evaluación izquierda a derecha con una pila
  • Sin necesidad de paréntesis ni precedencia
Expresión: a = b + c * d Polaca: b c d * + a =

2.2.2 Código P (P-Code)

  • Lenguaje intermedio independiente de la máquina
  • Diseñado para ejecutarse en una máquina virtual de pila
  • Usado originalmente en el compilador Pascal
a := b + c * d → LOAD b LOAD c LOAD d MUL ADD STORE a

2.2.3 Triples

  • Cada instrucción tiene 3 campos: (op, arg1, arg2)
  • El resultado se referencia por número de instrucción
  • No usa variables temporales explícitas
# | op | arg1 | arg2 0 | * | c | d 1 | + | b | (0) 2 | := | a | (1)

2.2.4 Cuádruplos

  • Cada instrucción tiene 4 campos: (op, arg1, arg2, resultado)
  • Usa variables temporales explícitas (t1, t2...)
  • Más flexible, permite reordenar instrucciones
# | op | arg1 | arg2 | result 0 | * | c | d | t1 1 | + | b | t1 | t2 2 | := | t2 | | a
cuádruplos triples código-P temporales
2.3
Esquema de Generación
Variables, expresiones, control, funciones
+

El esquema de generación define cómo se traduce cada estructura del lenguaje fuente a código intermedio. Cubre desde variables simples hasta estructuras complejas como funciones y arreglos.

2.3.1 Variables y constantes

  • Las variables generan instrucciones LOAD/STORE
  • Las constantes se usan directamente como operandos

2.3.2 Expresiones

  • Se generan en orden post-fijo recorriendo el AST
  • Se crean variables temporales para resultados intermedios

2.3.3 Instrucción de asignación

Fuente: x = a + b * c Cuádruplos: t1 = b * c t2 = a + t1 x = t2

2.3.4 Instrucciones de control

if (a < b) goto L1 // código else goto L2 L1: // código if L2: // continúa

2.3.5 Funciones

  • Instrucciones PARAM para pasar argumentos
  • Instrucción CALL con nombre y número de args
  • Instrucción RETURN para devolver valor

2.3.6 Estructuras (structs / arrays)

  • Acceso por desplazamiento (offset) desde dirección base
  • Fórmula array: base + i * tamaño_elemento
goto PARAM CALL offset temporales

3.1
Tipos de Optimización
Local, ciclos, global, mirilla
+

La optimización de código es el proceso de transformar el código intermedio para que sea más eficiente en términos de tiempo de ejecución y/o uso de memoria, sin alterar el resultado del programa.

3.1.1 Optimización Local (de bloque básico)

  • Actúa dentro de un solo bloque básico (secuencia lineal sin saltos)
  • Eliminación de subexpresiones comunes: no recalcular lo ya calculado
  • Propagación de constantes: sustituir variables con valores conocidos
  • Plegado de constantes: evaluar expresiones constantes en compilación
// Antes: t1 = a * b t2 = a * b // subexpresión repetida t3 = t1 + t2 // Después (optimizado): t1 = a * b t3 = t1 + t1 // reutiliza t1

3.1.2 Optimización de Ciclos

  • Movimiento de código invariante: sacar del ciclo lo que no cambia
  • Reducción de fuerza: reemplazar operaciones costosas (x²→x+x)
  • Eliminación de variables de inducción: simplificar contadores
  • Desenrollado de ciclos: expandir el cuerpo para reducir saltos
// Antes — código invariante dentro del ciclo: for i = 1 to n: x = a * b // a y b no cambian en el ciclo! arr[i] = x + i // Después — movimiento de código: x = a * b // fuera del ciclo for i = 1 to n: arr[i] = x + i

3.1.3 Optimización Global

  • Actúa sobre el flujo completo del programa
  • Requiere análisis de flujo de datos entre bloques
  • Propagación global de constantes entre bloques
  • Eliminación de código muerto (dead code elimination)

3.1.4 Optimización de Mirilla (Peephole)

  • Examina una ventana pequeña de instrucciones a la vez
  • Reemplaza secuencias ineficientes por equivalentes mejores
  • Eliminación de saltos redundantes
  • Simplificación algebraica: x + 0 → x, x * 1 → x
  • Instrucciones de máquina específicas más eficientes
// Mirilla — elimina salto redundante: Antes: JUMP L1 L1: ... Después: ... (elimina el JUMP innecesario) // Mirilla — simplificación algebraica: Antes: t1 = x * 2 Después: t1 = x + x (o usa shift: x << 1)
bloques básicos dead code ciclos peephole flujo de datos
3.2
Costos de Optimización
Ejecución, criterios, herramientas
+

Los costos en optimización se refieren al tiempo, espacio y recursos que consume cada instrucción del código intermedio y generado. El objetivo es minimizarlos sin cambiar el resultado del programa.

3.2.1 Costo de ejecución

  • Memoria: cuánta RAM consume el programa y sus estructuras
  • Registros: uso eficiente de registros del procesador (acceso más rápido)
  • Pilas: profundidad y operaciones en la pila de llamadas
  • Las instrucciones con operandos en registros son más baratas que en memoria

3.2.2 Criterios para mejorar el código

  • Tiempo de ejecución: minimizar ciclos de reloj consumidos
  • Tamaño del código: reducir número de instrucciones generadas
  • Uso de registros: maximizar reutilización, minimizar accesos a memoria
  • Localidad de referencia: mejorar caché hits
  • Hay un trade-off: código más rápido puede ser más grande (y viceversa)

3.2.3 Herramientas para análisis de flujo de datos

  • Grafos de flujo de control (CFG): representan rutas de ejecución
  • Análisis de alcance de definiciones: qué definiciones llegan a qué puntos
  • Análisis de variables vivas: qué variables se usan después de un punto
  • Análisis de disponibilidad: expresiones ya calculadas que están disponibles
// Análisis de variables vivas: // Si 'x' no se usa después de este punto → 'x' está muerta // Podemos eliminar la asignación a 'x' t1 = a + b x = t1 * 2 // ← si 'x' no se usa más → código muerto y = t1 + 5 // solo 'y' se necesita
CFG variables vivas registros flujo de datos trade-off

4.1
Registros
Asignación y uso de registros del procesador
+

Los registros son la unidad de almacenamiento más rápida del procesador. La asignación eficiente de registros es uno de los problemas más críticos en la generación de código objeto, ya que minimizar los accesos a memoria mejora dramáticamente el rendimiento.

¿Por qué importan los registros?

  • Son 10–100× más rápidos que acceder a memoria RAM
  • El número de registros es limitado (8–32 en arquitecturas comunes)
  • Si no alcanza un registro → se hace un spill (guardar en memoria)
  • Cada spill añade instrucciones LOAD/STORE innecesarias

Registros en arquitectura x86/x86-64

; Registros de propósito general (64-bit): RAX — acumulador, retorno de funciones RBX — base, preservado entre llamadas RCX — contador (ciclos, operaciones de cadena) RDX — datos, multiplicación/división RSI — índice fuente (source index) RDI — índice destino (destination index) RSP — puntero de pila (stack pointer) RBP — puntero base del marco (frame pointer) R8–R15 — registros adicionales (solo 64-bit)

Estrategias de asignación de registros

  • Asignación global: análisis de todo el flujo del programa (costosa pero óptima)
  • Asignación local: solo dentro de un bloque básico (rápida, menos óptima)
  • Coloración de grafos: modela conflictos entre variables como un grafo y lo colorea con k colores (k = número de registros)
  • Heurística LRFU: descarta el registro usado menos recientemente

Ejemplo — coloración de grafos

Variables vivas al mismo tiempo → no pueden compartir registro a ─── b Si hay 2 registros (R1, R2): │ │ a → R1 └─ c ─┘ b → R2 c → R1 (a ya terminó) ✓
x86 RAX/RBX spill coloración de grafos variables vivas
4.2
Lenguaje Ensamblador
Instrucciones, modos de direccionamiento, traducción
+

El lenguaje ensamblador (Assembly) es una representación simbólica del lenguaje máquina. Cada instrucción en ensamblador corresponde directamente a una instrucción de la CPU. El compilador genera ensamblador como paso previo al código binario final.

Estructura de una instrucción

[etiqueta:] mnemónico [operando1, operando2] [; comentario] Ejemplo: inicio: MOV EAX, 5 ; EAX = 5 ADD EAX, EBX ; EAX = EAX + EBX JMP inicio ; salta a etiqueta

Modos de direccionamiento

  • Inmediato: el operando es un valor constante — MOV EAX, 10
  • Registro: el operando está en un registro — MOV EAX, EBX
  • Directo: el operando es una dirección de memoria — MOV EAX, [0x1000]
  • Indirecto por registro: la dirección está en un registro — MOV EAX, [EBX]
  • Indexado: base + desplazamiento — MOV EAX, [EBX+4]

Traducción de código intermedio a ensamblador

; Código intermedio: ; Ensamblador x86: t1 = b * c → MOV EAX, [b] MUL [c] ; EAX = b * c MOV [t1], EAX t2 = a + t1 → MOV EBX, [a] ADD EBX, [t1] ; EBX = a + t1 MOV [t2], EBX x = t2 → MOV EAX, [t2] MOV [x], EAX

Instrucciones comunes x86

  • MOV: mover/copiar datos entre registros o memoria
  • ADD / SUB / MUL / DIV: operaciones aritméticas
  • CMP: comparar dos valores (afecta flags)
  • JMP / JE / JNE / JL / JG: saltos condicionales e incondicionales
  • PUSH / POP: operaciones en la pila
  • CALL / RET: llamada y retorno de subrutinas
MOV JMP CALL/RET mnemónicos direccionamiento
4.3
Lenguaje Máquina
Formato binario, codificación de instrucciones
+

El lenguaje máquina es la representación binaria final que ejecuta directamente el hardware. Es el último paso del proceso de compilación. El ensamblador traduce los mnemónicos a este formato binario mediante un proceso llamado ensamblado.

Formato de instrucción

  • Opcode (código de operación): identifica qué instrucción es (ADD, MOV, JMP…)
  • Operandos: registros, direcciones o valores inmediatos
  • Modo de direccionamiento: indica cómo interpretar los operandos
  • En x86 las instrucciones tienen longitud variable (1 a 15 bytes)

Ejemplo — codificación binaria

Ensamblador: MOV EAX, 5 Hexadecimal: B8 05 00 00 00 Binario: B8 = 1011 1000 → opcode MOV EAX, imm32 05000000 = 00000000 00000000 00000000 00000101 → valor 5 ; Otro ejemplo: ADD EAX, EBX → 01 D8 CMP EAX, 0 → 83 F8 00 JE etiqueta → 74 XX (XX = desplazamiento relativo)

Proceso completo: fuente → máquina

Código fuente (C/Pascal): x = a + b * c ↓ Análisis léxico, sintáctico, semántico Código intermedio (cuádruplos): t1 = b * c x = a + t1 ↓ Generación de código objeto Ensamblador: MOV EAX, [b] MUL [c] ADD EAX, [a] MOV [x], EAX ↓ Ensamblado Lenguaje máquina (hex): 8B 05 .. .. .. .. F7 25 .. .. .. .. 03 05 .. .. .. ..

Tipos de código objeto

  • Código reubicable: direcciones relativas, el linker las ajusta
  • Código absoluto: direcciones fijas, listo para ejecutar
  • Código compartido (shared libs): reutilizable por varios programas
opcode binario hex linker reubicable
4.4
Administración de Memoria
Pila, heap, segmentos, marcos de activación
+

La administración de memoria define cómo el compilador organiza el uso de la memoria en tiempo de ejecución. Es crítico para el correcto funcionamiento de variables locales, parámetros, valores de retorno y objetos dinámicos.

Segmentos de memoria de un programa

┌─────────────────────┐ ← dirección alta │ STACK (pila) │ variables locales, retornos │ ↓ │ crece hacia abajo ├─────────────────────┤ │ │ espacio libre ├─────────────────────┤ │ HEAP (montículo) │ memoria dinámica (malloc/new) │ ↑ │ crece hacia arriba ├─────────────────────┤ │ BSS │ variables globales no inicializadas ├─────────────────────┤ │ DATA │ variables globales inicializadas ├─────────────────────┤ │ TEXT (código) │ instrucciones del programa └─────────────────────┘ ← dirección baja (0x0000...)

Marco de activación (Activation Record / Stack Frame)

Cada llamada a función crea un marco de activación en la pila. Contiene toda la información necesaria para ejecutar y retornar de la función.

┌──────────────────────────┐ │ Parámetros de entrada │ ← pasados por el llamador ├──────────────────────────┤ │ Dirección de retorno │ ← a dónde volver al terminar ├──────────────────────────┤ │ RBP anterior (saved EBP) │ ← enlace dinámico ├──────────────────────────┤ │ Variables locales │ ← espacio para vars de la función ├──────────────────────────┤ │ Temporales │ ← resultados intermedios └──────────────────────────┘ ← RSP (tope de pila actual)

Estrategias de asignación

  • Estática: tamaño fijo en compilación — variables globales, constantes
  • Dinámica en pila: al llamar funciones — variables locales, parámetros
  • Dinámica en heap: malloc/new en tiempo de ejecución — objetos, arreglos dinámicos

Convenciones de llamada (Calling Conventions)

  • cdecl: argumentos en pila de derecha a izquierda, caller limpia
  • stdcall: igual pero callee limpia la pila (WinAPI)
  • fastcall: primeros argumentos en registros (ECX, EDX)
  • System V AMD64 ABI: Linux x86-64 — primeros 6 args en RDI, RSI, RDX, RCX, R8, R9
stack frame heap malloc cdecl ABI RSP/RBP
